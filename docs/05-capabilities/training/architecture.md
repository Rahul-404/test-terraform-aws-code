# Training Capability Architecture

## Purpose

This document describes the internal architecture of the Training Capability.

The goal of the architecture is to provide a standardized, reproducible, scalable, and observable mechanism for executing machine learning training workloads across all projects onboarded to the platform.

This document focuses on logical architecture rather than infrastructure implementation details.

Infrastructure mappings are documented separately within Terraform and Infrastructure documentation.

---

# Architectural Goals

The Training Capability is designed around the following principles:

* Reproducible execution
* Standardized workflows
* Infrastructure abstraction
* Operational simplicity
* Project independence
* Observable execution
* Future scalability

The capability should support multiple machine learning projects without requiring project-specific platform modifications.

# Architectural Components

The capability consists of several logical components.

Each component owns a specific responsibility.

---

# Training API

## Purpose

Acts as the entry point into the Training Capability.

Responsibilities:

* Accept training requests
* Validate requests
* Authenticate callers
* Authorize execution
* Create training jobs

The API provides a stable interface for platform consumers.

---

# Training Orchestrator

## Purpose

Coordinates the complete lifecycle of a training workload.

Responsibilities:

* Receive validated requests
* Manage workflow execution
* Trigger training environments
* Track execution progress
* Handle retries
* Process completion events

The orchestrator serves as the control plane of the capability.

---

# State Manager

## Purpose

Maintains lifecycle state for every training execution.

Responsibilities:

* State transitions
* Status tracking
* Failure tracking
* Execution history

Example states:

* Submitted
* Queued
* Running
* Completed
* Failed
* Cancelled

The State Manager is the authoritative source of execution status.

---

# Metadata Layer

## Purpose

Captures metadata required for reproducibility and governance.

Responsibilities:

* Dataset references
* Container versions
* Configuration values
* Runtime information
* Execution timestamps

Metadata generated by this component is consumed by:

* Experiment Tracking
* Governance
* Model Registry

---

# Resource Layer

## Purpose

Abstracts infrastructure allocation from platform consumers.

Responsibilities:

* Compute allocation
* Resource configuration
* Resource cleanup
* Capacity management

Consumers should never interact directly with compute resources.

---

# Training Compute

## Purpose

Executes model training workloads.

Responsibilities:

* Container execution
* Dataset loading
* Feature retrieval
* Model training
* Validation
* Artifact generation

Training Compute represents the data plane of the capability.

---

# Artifact Storage

## Purpose

Stores outputs generated during training execution.

Examples:

* Models
* Checkpoints
* Evaluation reports
* Feature statistics
* Training logs

Artifacts must be immutable and versioned.

---

# Integration Layer

## Purpose

Publishes outputs to downstream platform capabilities.

Consumers include:

## Experiment Tracking

Receives:

* Parameters
* Metrics
* Artifacts
* Run metadata

---

## Model Registry

Receives:

* Candidate models
* Validation results
* Model metadata

---

## Monitoring

Receives:

* Job metrics
* Runtime metrics
* Failure metrics

---

## Governance

Receives:

* Lineage information
* Audit metadata
* Ownership metadata

---

# Control Plane vs Data Plane

The architecture intentionally separates orchestration from execution.

---

## Control Plane

Responsible for:

* Request handling
* Scheduling
* State management
* Metadata management
* Coordination

Components:

* Training API
* Training Orchestrator
* State Manager
* Metadata Layer

The Control Plane decides what should happen.

---

## Data Plane

Responsible for:

* Training execution
* Data processing
* Artifact generation

Components:

* Training Compute
* Artifact Storage

The Data Plane performs the actual work.

---

# Capability Boundaries

The architecture begins when a valid training request enters the capability.

The architecture ends when:

* Artifacts are produced
* Metadata is recorded
* Downstream capabilities are notified

Responsibilities beyond this point belong to:

* Experiment Tracking
* Model Registry
* Deployment
* Monitoring
* Governance

| Training Compute       | SageMaker Training Jobs |
| Artifact Storage       | S3                      |
| Monitoring Integration | CloudWatch              |
| Scheduling Integration | EventBridge             |

This mapping may evolve as platform requirements change.

# Architectural Characteristics

The architecture provides:

* Reproducible training
* Infrastructure abstraction
* Standardized execution
* Loose coupling
* Clear ownership boundaries
* Operational visibility
* Future extensibility

The architecture is intentionally optimized for startup-scale machine learning workloads rather than enterprise-scale distributed training systems.

---

# Summary

The Training Capability architecture separates orchestration concerns from execution concerns through a clear control plane and data plane design.

This enables the platform to provide a standardized training experience while maintaining reproducibility, operational simplicity, and clean integration with experiment tracking, model registry, monitoring, deployment, and governance capabilities.
